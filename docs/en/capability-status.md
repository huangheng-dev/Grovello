# Capability status

| Status | Meaning |
| --- | --- |
| Foundation | Route, product definition, shared objects, architecture boundary, and initial contract exist |
| Simulated | Demonstration uses labeled seed or mock data and does not make external claims |
| Connected | A provider account has passed contract, auth, health, policy, and sandbox tests |
| Operational | End-to-end execution, failure handling, audit, measurement, and runbook are verified |
| Experimental | Usable behind an explicit flag with known limits and evaluation criteria |
| Planned | Accepted architecture item with no production implementation claim |

Current repository status: Phase 0 is verified. Phase 1 organization and access is in foundation implementation: canonical organization, workspace, user, team, membership, role, permission, policy, session, and audit models exist; the PostgreSQL migration enables workspace RLS; and seeded access contracts verify authenticated, unauthorized, cross-tenant, audit, and recovery-plan paths. Phase 2 shared business truth now has a foundation slice for canonical object IDs, immutable versions, exact evidence citations, profile validation, idempotent writes, audit lineage, transactional outbox events, and workspace RLS. Its contract covers brand, product, offer, price book, market, ICP, evidence, knowledge, assets, and cases, while the Northstar data remains a labeled fictional fixture. The six Brand & Market pages now provide canonical profile reads, governed object creation, and immutable version updates through a server-side BFF, including loading, empty, unavailable, unauthorized, conflict, and validation states. Business & Brand, Products, Offers & Pricing, Markets & Localization, Ideal Customer Profiles, Evidence, and Knowledge Documents provide localized domain forms. Offers reference canonical product IDs; price books can reference canonical offer IDs and store typed price entries; ICPs reference canonical market IDs and optional product IDs while storing structured buying committees, qualification criteria, and exclusions. Evidence records capture provenance, verification status, reliability, usage rights, scope, findings, and limitations. Knowledge documents can reference one canonical subject and store governed content, ownership, topics, audiences, and retrieval keywords. Case studies now capture disclosure status, approved customer naming, canonical market/product/offer/ICP links, engagement dates, challenge, approach, structured measured outcomes, limitations, approved uses, authorization reference, and ownership; an active case must cite at least one exact immutable evidence version. Version editors can pin supported claims to exact immutable evidence versions with structured locators, and the knowledge page can filter the records already retrieved from the live workspace API. Unknown attributes from earlier versions are retained during edits. The Asset Library storage foundation now defines tenant-scoped upload sessions, verified immutable blobs, exact asset-version file bindings, workspace RLS, and risk-tiered asset permissions. The local reference S3-compatible profile is connected through fixed MinIO images, a private versioned bucket, a least-privilege application account, bounded provider-neutral operations, and an explicit health endpoint. Upload-session APIs now provide idempotent creation, exact size/type/SHA-256 constrained presigned POST grants, authorized status/complete/cancel operations, audit and outbox lineage, and a durable Temporal verification workflow. Verification checks provider metadata and streams stored bytes to calculate SHA-256. A provider-neutral malware-scanner contract and digest-pinned ClamAV local reference service now stream verified objects through bounded Temporal activities. Clean sessions advance only to `ready_to_finalize`; infected objects are copied to a tenant-scoped quarantine key, removed from staging by exact version, and recorded with scanner evidence. Exhausted or non-retryable failures fail closed, and cancellation uses native Temporal cancellation plus the persisted session state. No Ready Asset, immutable Blob, or asset-version binding is created by scanning. Governed finalization is now connected through an idempotent Temporal Saga: only clean sessions are promoted to immutable workspace-scoped object versions, the canonical Asset/version/file binding is committed transactionally, exact staging versions are cleaned, and failed commits are safely compensated. Active finalization requires `asset.approve`; audited short-lived private downloads require `asset.download` and an exact active, clean, available version. The Asset Library now provides a localized operational foundation UI through a server-side BFF: governed browser upload, durable status tracking, clean-session finalization, canonical catalog reads, immutable version history, and audited secure download authorization are connected without exposing object keys or storage credentials. It remains a `foundation` capability until production identity and the general approval workflow are connected. Persistent owner onboarding, imports, semantic retrieval, knowledge chunking pipelines, full approval workflows, and provider synchronization are also not yet operational; knowledge chunks remain derived pipeline objects rather than owner-authored records. Development identity is accepted by the BFF only behind an explicit local switch, the API still rejects development headers in production, and no OIDC provider or persistent access directory is connected yet. External channels, analytics projections, and production provider accounts are not connected.

P2-D1 through P2-D4 now provide a verified local foundation for persistent workspace onboarding and structured business-truth imports: tenant-scoped execution tables with forced RLS, nine risk-tiered permissions, idempotent onboarding/import APIs, constrained private source upload, dedicated Temporal integrity/scanning/validation/apply/compensation workflows, bounded UTF-8 CSV and versioned Grovello JSON parsing, immutable allowlisted mappings, deterministic canonical validation, exact-only deduplication, redacted review evidence, immutable dry-run change sets, policy-versioned approval handoff, canonical business-truth writes, concurrency-safe compensating versions, and exact activation snapshots with `BusinessProfileActivated` lineage. P2-D4 adds localized Business Setup and Imports operator routes through the thin BFF, persistent progress and recovery states, accessible forms, and responsive layouts. The July 23 local acceptance run used PostgreSQL, Temporal, a private versioned MinIO bucket, its least-privilege application account, and ClamAV to upload, verify, scan, map, validate, plan, apply, and compensate a browser-originated CSV; the job produced nine audit records and nine outbox events. A separate fresh-workspace integration run verified exact activation and compensation. The capability remains `foundation`, not `operational`, until production identity, the general approval workflow, production runbooks, and non-development deployment evidence are connected.

P2-E1 now provides a locally verified enterprise-knowledge ingestion contract foundation. Migration `0012` adds five tenant-scoped execution tables with forced RLS, seven risk-tiered knowledge permissions, immutable exact-source snapshots, pending generations, and retrieval-receipt storage reserved for later slices. The versioned API supports idempotent create/list/read ingestion operations for eligible active source versions; internal service actors alone may create canonical immutable `knowledge_chunk` versions with exact source and generation lineage, while generic owner/import mutation and generic business-truth reads cannot bypass that boundary. Verification covered real PostgreSQL migration parity, non-superuser cross-workspace RLS isolation, HTTP authorization/idempotency/conflict behavior, pipeline replay safety, and exact audit/outbox evidence. P2-E1 remains `foundation`: parsing, Temporal extraction, deterministic chunking, embeddings, lexical or semantic retrieval, citations, and an operator UI are not implemented.

## Navigation capability states

The navigation catalog retains all ten product domains and the stable identity of future capabilities. The primary sidebar shows only `active` capabilities with a working experience and `foundation` capabilities with an honest implementation boundary. A `reserved` capability remains visible in its domain capability map with its delivery phase and intended audience, but it is not a clickable primary-navigation feature and makes no operability claim. System Architecture is help content rather than a business module; advanced automation controls are reserved for administrator or developer experiences.

English navigation uses established global B2B SaaS and revenue-operations terminology. Simplified Chinese is localized by business meaning rather than translated word for word. Examples include Strategy & Plans / 战略与计划, Quote-to-Cash / 报价到回款, Unified Data / 统一数据, and AI Search (GEO) / AI 搜索优化（GEO）.
