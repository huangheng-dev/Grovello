# 企业知识入库与可引用检索实施基线

状态：**已批准；P2-E1 已实现并通过截至 2026 年 7 月 23 日的本地验证；P2-E2 至 P2-E4 尚未实现，不代表可运营**。

本基线定义 P2-E 的交付边界：把受治理企业知识转化为版本化、工作区隔离且可精确引用的检索能力。它补齐阶段 2“共享业务事实”的检索部分，不改变十个一级业务域、主导航或正式技术基线。

## 1. 必须对齐的变更要素

| 要素 | P2-E 对齐方式 |
| --- | --- |
| 目标 | 把已批准知识来源转化为不可变派生片段，并返回绑定业务目的且带精确来源版本引用的检索结果。 |
| 产品域 | “品牌与市场”拥有企业知识；“数据与智能”可消费检索证据；“自动化运行时”提供持久化工作流和模型路由契约。 |
| 共享对象 | `KnowledgeDocument`、`KnowledgeChunk`、`Asset`、`Evidence`、`CaseStudy`、`Citation`、`Run`、`Task`、`ModelConfig`、`AuditEvent`。 |
| 输入 | 精确的有效来源版本、语言、使用权、敏感等级、业务目的、流水线版本、解析器 Profile 和可选 Embedding 模型配置。 |
| 受治理动作 | 快照、提取、规范化、分类、切块、向量化、发布、检索、重建索引、取消、退役和重建投影。 |
| 输出 | 不可变知识 Generation、派生片段版本、可重建的词法/向量投影、检索回执、引用、审计事件和 Outbox 事件。 |
| 反馈路径 | 无结果、低分、来源过期、引用使用、延迟和操作员反馈用于修正来源、重建索引并提升后续策略质量。 |
| 权限与审批 | 窄粒度知识权限、来源对象授权、敏感策略、目的绑定，以及高风险重建或退役动作审批。 |
| 验证 | RLS、迁移、解析限制、幂等、工作流重试/取消/补偿、检索隔离、引用准确、注入防护、双语体验和重建测试。 |

## 2. 产品与导航边界

- P2-E 深化“品牌与市场”中现有的**知识文档**体验。
- 不新增一级业务域，也不新增主导航入口。
- 入库状态、来源历史和检索测试界面可以放在现有知识路由中；只有页面过于密集时才允许使用高级子路由。
- `KnowledgeChunk` 始终是派生对象。所有者不能通过通用业务事实 Mutation API 或导入功能创建、编辑知识片段。
- P2-E 返回证据和引用，不生成营销文案、建议、策略、答案或自主 Agent 动作。
- “内容与流量”“数据与智能”和后续 Agent 可以消费公开检索契约，但不得建立私有片段副本。

## 3. 事实来源与投影规则

PostgreSQL 继续作为来源快照、入库状态、有效 Generation 选择、规范片段身份、检索回执、审计和 Outbox 血缘的事实来源。

`KnowledgeChunk` 使用现有规范业务对象身份和流水线创建的不可变版本。内部来源类型可以新增 `pipeline`，但公开的所有者/导入 Schema 必须继续拒绝 `source_type=pipeline` 和所有者创建的 `knowledge_chunk`。

词法检索结构和 pgvector Embedding 都是可重建投影，不能成为片段正文、来源定位、权限元数据或有效 Generation 状态的唯一副本。P2-E 不引入 OpenSearch。

每个片段必须解析到：

1. 一个精确来源对象及不可变来源版本；
2. 一个入库 Generation 和流水线版本；
3. 章节、标题、段落或页码等稳定来源定位；
4. 内容哈希、语言、使用权状态和敏感等级；
5. 生成该片段的解析器、切块器和 Embedding 血缘。

## 4. 初始交付范围

### 纳入治理的来源

- 已批准且有效的 `KnowledgeDocument` 版本；
- 使用权允许当前业务目的的已核验或所有者证明的有效 `Evidence` 版本；
- 已批准且有效的 `CaseStudy` 版本；
- 明确绑定为知识来源的精确、有效、干净且可用的 Asset 版本。

结构化 Brand、Product、Offer、PriceBook、Market 和 ICP 事实继续通过规范业务事实 API 获取。P2-E 可以筛选或引用这些对象，但不能把它们复制到不受控制的私有副本。

### 初始提取器策略

- 规范 `KnowledgeDocument` 正文是第一种必须支持的来源。
- UTF-8 纯文本和 Markdown Asset 可以通过同一版本化提取器契约继续接入。
- 支持可提取文本的 PDF 和 DOCX 前，必须确定固定版本解析器，完成许可证审查、有界解压、测试 Fixture 覆盖，并在 P2-E 范围内获得产品所有者明确批准。
- OCR、图片/音频/视频转写、电子表格、压缩包、可执行格式、加密文档、宏、远程 URL 抓取和外部爬取不属于初始验收门禁。
- 提取期间绝不执行或访问内嵌链接、脚本、附件和远程资源。

### 默认有界配置

实现设置统一使用 `GROVELLO_KNOWLEDGE_` 命名空间，并必须为以下内容设置正数上限：

- 来源字节、提取字符、页数、表格单元格和嵌套文档部件；
- 每个来源的片段数、目标片段大小、重叠和元数据字节；
- 查询字符、请求结果数、候选数和检索超时；
- 并发入库 Activity、解析时间、模型时间、重试次数和失败 Generation 保留数量。

默认值必须适配本地 Compose Profile。操作员可以按工作区策略降低限制；提高硬安全上限必须显式配置部署环境。

## 5. 规范执行记录

### KnowledgeIngestion

处理一个精确来源版本的一次幂等请求。记录工作区、操作者、目的、来源类型/ID/版本、请求的流水线 Profile、审批状态、状态、Temporal Workflow ID/Run ID、取消、失败分类、成本和时间戳。

### KnowledgeSourceSnapshot

工作流执行前解析的不可变输入封装。记录精确来源与 Asset 版本 ID、内容/Blob 哈希、语言、来源状态、使用权、敏感等级、解析器资格和授权所使用的策略版本。

### KnowledgeGeneration

一个来源快照与流水线版本对应的一次不可变输出 Generation。记录提取器、规范化器、分类器、切块器、Embedding 配置、数量、警告、发布状态和精确片段版本 ID 集合。一个来源版本和检索 Profile 最多只能有一个有效 Generation。

### KnowledgeChunk

由流水线创建的规范 `knowledge_chunk` 对象及不可变版本。其有界 Payload 包含规范化正文、顺序、来源定位、内容哈希、语言、Token/字符数量、主题、受众、使用权、敏感等级、不可信内容信号，以及精确的 Generation/来源血缘。

片段绝不静默更新。重新处理会发布新 Generation，并让旧 Generation 退出检索，同时保留历史。

### KnowledgeEmbeddingProjection

按工作区、片段版本、模型配置/版本、维度和内容哈希确定的可重建记录。向量只对该精确组合有效；维度变化或模型修订必须产生新投影 Generation。

### RetrievalReceipt

有界执行记录，包含工作区、操作者、目的、查询哈希、安全的规范化查询元数据、筛选条件、策略版本、词法/向量配置、返回片段版本 ID、分项分数、排名、延迟和时间戳。默认不把原始查询复制到审计证据；确需持久化时遵守工作区保留策略。

## 6. API 与服务边界

版本化 FastAPI 接口统一位于 `/api/v1/knowledge`：

- `POST /ingestions` 为精确来源版本创建幂等入库请求；
- `GET /ingestions` 列出有权限查看的有界摘要；
- `GET /ingestions/{ingestion_id}` 返回持久化进度、警告和失败状态；
- `POST /ingestions/{ingestion_id}/cancel` 请求 Temporal 原生取消；
- `POST /ingestions/{ingestion_id}/retry` 从可恢复终态显式创建重试；
- `POST /sources/{source_version_id}/reindex` 创建新的不可变 Generation；
- `POST /generations/{generation_id}/retire` 让 Generation 退出检索但不擦除血缘；
- `POST /retrievals` 执行有界且已授权的搜索，并返回带精确引用的检索回执；
- `GET /retrievals/{receipt_id}` 读取有权限查看且仍在保留期内的回执。

API 负责解析身份、授权、来源资格、幂等、策略和请求血缘。它绝不在 HTTP 请求内解析文档、调用 Embedding 供应商或执行无界索引。

授权后的有界 PostgreSQL 检索可以同步执行。入库、提取、Embedding、重建索引、扇出退役和重建投影均通过 Temporal Worker 运行。

Next.js 应用只使用同源薄 BFF。浏览器代码绝不会收到对象存储 Key、供应商凭据、无限制片段导出或开发身份 Header。

## 7. 持久化入库工作流

Temporal 负责确定性、版本化工作流：

```text
授权精确来源快照
→ 解析规范正文或已核验 Asset 绑定
→ 使用版本化有界解析器提取
→ 在不改变事实含义的前提下规范化
→ 分类语言、敏感等级和不可信内容信号
→ 创建确定性片段与定位
→ 在检索不可见状态下暂存不可变片段与词法投影
→ 通过供应商中立 Model Router 契约请求并暂存 Embedding
→ 在事务中发布有效 Generation 指针、审计和 Outbox
→ 退役上一代有效投影
```

片段与投影写入可以使用有界批次，但在最终有效 Generation 指针提交以前，检索不能看到其中任何内容。每个 Activity 都必须具有有界超时、错误分类、重试策略、必要的 Heartbeat 和确定性幂等键。取消会停止新工作，并且只清理由本次入库产生但尚未发布的投影。发布失败绝不能隐藏上一代有效 Generation。

LangGraph 不编排该流水线。后续 Agent 只能通过有界工具契约，把检索结果作为不可信证据消费。

## 8. Model Router 与检索契约

P2-E 只引入正式 Model Router 边界所需的最小公开 Embedding 能力契约。领域代码依赖规范化的 Embedding 请求与结果，绝不依赖某一家供应商 SDK。

契约记录供应商引用、模型 ID 与修订版、向量维度、Tokenizer/切块器兼容性、数据使用策略、区域策略、超时、成本和健康状态。Secret 始终是外部引用。

确定性 Fake Embedding 可以验证契约，但不能通过语义质量验收。真实本地或已配置供应商必须标注许可证、数据策略和能力状态。未配置 Embedding 供应商时，词法检索仍可使用，并在响应中明确返回 `semantic_status=unavailable`，不能伪造向量分数。

检索顺序为：

1. 应用工作区、来源状态、有效 Generation、语言、使用权、敏感等级、受众和目的筛选；
2. 获取 PostgreSQL 词法候选；
3. 仅在存在兼容投影时获取 pgvector 候选；
4. 使用确定性混合排名，并分别暴露词法与向量分数；
5. 使用稳定的同分排序和有界结果数量。

每个结果都包含精确片段版本、来源对象/版本、Generation、定位、安全摘要、分项分数和策略决定。P2-E 不在检索接口内生成自然语言答案。

## 9. 权限与审批

P2-E 新增窄权限，不扩大 `business_truth.read` 或 `business_truth.write`：

| 权限 | 风险 | 用途 |
| --- | --- | --- |
| `knowledge.retrieve` | R0 | 在操作者和业务目的已有权访问的来源上执行有界检索。 |
| `knowledge.ingest` | R1 | 入库符合条件的已批准来源版本。 |
| `knowledge.reindex` | R1 | 在不修改来源的情况下创建新 Generation。 |
| `knowledge.cancel` | R1 | 取消操作者拥有或有权治理的入库任务。 |
| `knowledge.retire` | R2 | 让有效 Generation 退出检索。 |
| `knowledge.sensitive.read` | R2 | 检索高于普通工作区级别的敏感内容。 |
| `knowledge.admin` | R2 | 修改已批准流水线/模型 Profile 或启动工作区级重建。 |

除知识权限外，始终必须评估来源对象授权。检索权限绝不能绕过业务事实、Asset、使用权、区域、保留或敏感策略。

已经批准的来源在已批准且未变化的流水线下，可以按策略常规入库。新解析器/Profile、敏感等级下调、高影响退役或批量重建必须记录审批策略版本和审批决定。

## 10. 安全与隐私审查

- 提取内容是不可信数据，绝不是系统指令。
- 检索不能执行工具、访问链接、运行宏、抓取远程资源或修改策略。
- Prompt Injection 指标只作为信号保存，不能自动证明恶意，也不能赋予内容任何权限。
- 解析进程必须执行格式白名单、字节/页数/部件/时间限制、有界解压和禁止网络出口。
- 二进制来源必须已经通过 Asset 契约，且处于精确、有效、干净、可用状态。
- 所有租户入库、Generation、投影和回执表均强制工作区 RLS。
- 查询和片段在日志、Trace、审计证据和错误消息中执行最小化处理。
- 敏感结果要求显式授权，并从通用预览和导出中排除。
- 片段正文、Embedding 和回执遵守来源保留与删除策略。退出检索必须立即生效；物理删除继续使用受审计工作流。
- 跨租户缓存键、向量查询、模型批次和 Worker Activity 都必须携带并核验工作区范围。
- 模型供应商只能在已批准数据使用策略下接收最小必要的授权正文和声明元数据。

## 11. 可观测性与运维证据

必须采集入库时长、提取失败、片段数量、Embedding 成本、投影延迟、检索延迟、无结果率、被策略过滤结果数、过期 Generation、取消、重试和引用使用反馈。

日志与 Trace 关联请求 ID、工作区、入库任务、Workflow、来源版本、Generation、模型配置和回执，但不能暴露 Secret 或无限制正文。

运行手册必须覆盖入库卡住、解析失败、供应商中断、维度不匹配、投影损坏、重建索引、退役、恢复和从 PostgreSQL 事实来源完整重建投影。

## 12. 交付切片

### P2-E1 — 契约、隔离与规范派生身份

- 迁移、RLS、权限、入库/Generation/回执 Schema、仅限内部流水线创建片段、API 契约、幂等、审计、Outbox 和迁移一致性；
- 通用所有者与导入 API 继续拒绝创建 `knowledge_chunk`；
- 不宣称解析、Embedding 或语义能力。

实现证据：迁移 `0012` 创建 5 张带强制 RLS 的工作区知识执行表，并增加 7 项分风险权限。版本化 API 只允许对精确且符合条件的 active 来源版本创建和读取幂等 pending 入库；只有内部服务 Actor 可以创建带精确来源与 Generation 血缘的不可变流水线派生片段版本。本地验证覆盖真实 PostgreSQL 升级/check/降级/再升级一致性、非超级用户跨工作区 RLS 隔离、HTTP 幂等/冲突/授权行为、流水线重放安全、通用 API 隐藏，以及精确审计与 Outbox 证据。

### P2-E2 — 持久化提取与确定性切块

- Temporal 工作流、规范正文入库、有界提取器注册表、规范化、分类、确定性定位/切块、取消、重试、补偿和上一代 Generation 安全；
- 初始二进制提取器只有在依赖、许可证和对抗性解析测试通过后才能接入；
- 不宣称语义质量。

### P2-E3 — 供应商中立 Embedding 与可引用检索

- 最小 Model Router Embedding 契约、pgvector 投影、PostgreSQL 词法检索、混合排名、策略筛选、检索回执、精确引用、供应商不可用降级、评估 Fixture 和重建测试；
- 不生成答案，也不提供 Agent 自主执行。

### P2-E4 — 双语操作体验与验收

- 在现有知识体验中提供入库状态/历史、警告、重试/取消/重建/退役控件、检索测试、引用/来源检查、响应式可访问性和中英文一致性；
- 完成从已批准来源到检索、精确引用、重建索引、退役、审计/Outbox 血缘和投影重建的完整本地多服务验收。

## 13. 完成门禁与非声明

只有一个干净工作区能入库已批准的精确来源版本、发布确定性片段、检索带精确引用的有权限结果、通过重试/取消恢复、安全替换和退役 Generation、重建投影，并在两种语言中复现且无跨租户泄露，P2-E 才算完成。

在生产身份、通用审批、生产运行手册、非开发部署和已批准真实 Embedding Profile 完成验证以前：

- 知识入库与检索仍是 `foundation`，不能标记为 `operational`；
- 确定性 Fake Embedding 只用于测试；
- 不宣称语义相关性、答案正确性或业务结果提升；
- 不宣称 OCR、转写、爬取、无限制文件解析、Agent 答案生成或外部供应商同步。

## 14. 架构、安全与交付审查结论

该提案与当前宪章兼容：

- FastAPI 负责领域契约，PostgreSQL 负责规范状态；
- Temporal 负责持久化入库和重建工作流；
- pgvector 与词法索引保持为可重建投影；
- Model Router 边界避免供应商锁定；
- LangGraph 是后续的有界消费者，不是入库编排器；
- 不改变产品域、主导航或正式技术基线。

批准本基线不需要 ADR。若实现方案提出新的强制数据库/搜索服务、把 OpenSearch 设为必需、在领域代码中嵌入供应商 SDK、修改十个业务域或主导航、允许不受控制的远程提取，或把工作流持久性从 Temporal 转移到 LangGraph，则必须先建立 ADR。
