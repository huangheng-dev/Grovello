# 工作区入驻与结构化导入实施基线

状态：**已批准；P2-D1 至 P2-D4 基础能力已实现，并已于 2026 年 7 月 23 日完成本地验证**。

本基线定义 P2-D 的交付边界：持久化工作区入驻，以及结构化业务事实导入。它落实产品交付路线图的阶段 2，不改变十个一级业务域和正式技术基线。

## 1. 必需的变更对齐

| 事项 | 基线 |
| --- | --- |
| 目标 | 让工作区所有者创建、校验、审查、激活并持续完善一份规范企业资料，包括从结构化文件进行受治理批量摄取。 |
| 主要产品域 | 品牌与市场。组织与治理只负责工作区设置、成员、策略、权限和审计。 |
| 共享对象 | Workspace、Brand、Product、Offer、PriceBook、Market、ICP、Evidence、KnowledgeDocument、Asset、CaseStudy、BusinessObjectVersion、Citation、Approval、Run、AuditEvent。 |
| 输入 | 所有者编辑、UTF-8 CSV、带版本的 Grovello JSON 导入包、精确规范对象 ID 和不可变证据版本 ID。 |
| 动作 | 上传、核验、扫描、解析、映射、规范化、校验、去重、预览、审批、应用、取消、补偿和激活。 |
| 输出 | 规范对象草稿或不可变对象版本、精确引用、导入结果与问题报告、入驻完整度、审计事件和事务 Outbox 事件。 |
| 反馈路径 | 校验缺口和导入结果更新入驻完整度；已激活业务事实成为情报、战略、执行、归因以及下一轮战略的受治理输入。 |
| 权限与审批 | 导入执行与资料激活分离。应用 active 版本、解决破坏性冲突或补偿已接受变更时，必须经过策略评估和高权限控制。 |
| 验证 | Schema 与迁移、工作区 RLS、权限与跨租户、解析安全、幂等与工作流重放、审计/Outbox、双语界面、端到端导入与补偿测试。 |

## 2. 产品与导航边界

工作区入驻是跨现有规范能力的流程，不是第十一个产品域，也不会替代“品牌与市场”的六个模块。

- 工作区设置体验可以进入持久化入驻清单，并恢复上次未完成步骤。
- 能力实现后，“品牌与市场”可以提供“导入”操作子路由，并在相关页面提供上下文“导入”入口。
- 用户界面中，英文使用 **Business Setup** 表示这条资料流程、使用 **Imports** 表示导入工作台；简体中文分别使用 **企业资料配置** 和 **数据导入**。技术术语 Workspace Onboarding 不得与“客户增长”域售后阶段的 **Onboarding & Success / 客户入驻与成功** 混淆。
- 导入不得成为新的一级业务域，也不得变成与业务事实脱节的数据管理产品。
- 未来通用摄取可观测性和跨域导入/导出治理可以归属“数据与智能”，但它不拥有或复制“品牌与市场”的业务事实。
- P2-D 不改变一级导航。只有端到端契约、失败状态、权限和双语体验通过验证后，路由才能标记为 active。
- 同一入驻和导入契约必须能服务另一种 B2B 产品、服务、软件方案、来源国和目标市场，不得修改规范模型。

## 3. 事实来源规则

导入作业是受治理的摄取与变更计划记录，绝不能成为品牌、产品、方案、价格、市场、ICP、证据、知识、素材或案例的竞争性存储。

已接受行必须调用与所有者编辑相同的 FastAPI 应用服务，创建规范 `BusinessObject` 身份及 `source_type=import` 的不可变 `BusinessObjectVersion`。每个版本保留导入作业、映射版本、来源校验和、操作者、业务目的、幂等键、输入版本和精确引用血缘。

PostgreSQL 仍是事务事实来源。对象存储只在有限保留期内保存私有来源文件和生成的报告。Temporal 历史、缓存、预览和暂存行都是执行状态，不能替代规范记录。

## 4. 首期交付范围

### 包含的对象类型

第一实现切片接受：

- 品牌、产品、商业方案、价格表、市场和 ICP；
- 证据、知识文档和客户案例；
- 对现有规范素材和精确证据版本的引用。

素材二进制继续使用素材库的上传、核验、扫描、最终化和权限契约。P2-D 不建立第二条二进制上传通道。知识片段仍是派生流水线对象，不允许所有者导入。

### 包含的来源格式

- 使用显式分隔符和表头行的 UTF-8 CSV；
- 带版本的 Grovello JSON 导入包，其 Manifest 声明 Schema 版本、语言、对象类型和记录数。

XLSX、外部 URL、供应商网盘、压缩包、宏、嵌入文件和远程同步不在第一实现切片内。后续可在依赖、许可证、安全和资源限制评审通过后，通过解析器适配器增加 XLSX；它必须生成同一套规范化暂存契约。

### 默认资源限制

默认值可通过 `GROVELLO_IMPORT_` 命名空间按部署配置，并由 API 与 Worker 同时执行：

- 来源文件最大 25 MiB；
- 每项作业最多 10,000 条记录；
- 每条 CSV 记录最多 100 列；
- 每个标量值最多 64 KiB；
- JSON 嵌套深度最大 12；
- 每个工作区同时只允许一个 apply 工作流，校验并发有可配置上限。

提高限制必须提供负载证据，重新评审超时与内存，并更新运维手册。HTTP 请求绝不内联解析或应用批量数据。

## 5. 规范执行记录

所有租户记录均包含 `workspace_id`、时间戳、操作者与请求血缘，并启用工作区 RLS。

### WorkspaceOnboarding

每个工作区一条持久化入驻记录，状态为 `draft`、`in_progress`、`ready_for_review`、`active` 或 `blocked`；记录必需对象类型、校验缺口、策略版本、最后完成步骤、激活版本以及激活操作者/时间。它只保存完成证据，不私存业务数据副本。

### ImportJob

作业记录业务目的、来源格式与校验和、对象类型、解析器与 Schema 版本、所选映射版本、状态、行数统计、Dry-run 计划哈希、Temporal Workflow/Run ID、幂等键、保留期限、取消/补偿状态和脱敏结果摘要。

作业状态为：

```text
created -> uploading -> verifying -> scanning -> mapping -> validating
-> ready_for_review -> applying -> completed | partially_completed
```

终态及恢复状态包括 `failed`、`cancelled`、`expired`、`compensating` 和 `compensated`。重试从持久化状态恢复，不创建第二个逻辑作业。

### ImportSource

来源记录把一项作业绑定到精确的私有对象存储 Key 与供应商版本，并记录声明/核验后的字节长度、类型、校验和、扫描结果、加密状态和删除期限。它复用素材流水线已经使用的供应商中立存储、核验和恶意文件扫描适配器，但不会创建规范 Asset，也不会建立第二套业务数据存储。存储标识只在内部使用，绝不返回浏览器客户端。

### ImportMappingVersion

映射一经使用即不可变。每个版本记录来源列选择器、目标字段、确定性转换、语言与类型转换、默认值、引用解析规则、创建者和 Schema 指纹。禁止 Secret、可执行表达式、网络调用和任意代码。

### ImportRow 与 ImportIssue

暂存行保存受限的规范化表示、来源行号、内容哈希、目标身份、校验状态、计划操作、已应用对象/版本 ID 和错误代码。问题记录只含安全字段定位器和脱敏消息，不保存无限制来源单元格。

行状态为 `pending`、`valid`、`invalid`、`duplicate`、`conflict`、`planned`、`applied`、`skipped`、`failed` 和 `compensated`。

### ImportChangeSet

不可变 Dry-run 计划记录每项 create、new-version、skip 或 conflict 决策，以及精确的预期当前版本。规范输入在审查后发生变化时，apply 必须拒绝或重新规划。Change Set 是审批、审计、取消和补偿的单位。

## 6. API 与服务边界

FastAPI 负责带版本业务契约和鉴权。Next.js 只负责本地化交互、会话上下文和同源薄 BFF。

计划中的 API 分组为：

```text
/api/v1/workspace-onboarding
/api/v1/workspace-onboarding/activate
/api/v1/import-jobs
/api/v1/import-jobs/{job_id}/source-upload
/api/v1/import-jobs/{job_id}/mapping
/api/v1/import-jobs/{job_id}/validation
/api/v1/import-jobs/{job_id}/change-set
/api/v1/import-jobs/{job_id}/change-set/approval
/api/v1/import-jobs/{job_id}/apply
/api/v1/import-jobs/{job_id}/cancel
/api/v1/import-jobs/{job_id}/compensate
```

每个 Mutation 都要求 `Idempotency-Key`、业务目的、已授权工作区上下文，以及相关情况下的乐观输入版本。列表和详情响应不得暴露对象存储 Key、凭据、无限制来源行或其他工作区数据。

## 7. 持久化工作流

Temporal 负责长时导入生命周期：

1. 授权创建作业，并签发受约束的私有来源上传授权。
2. 核验精确对象版本、预期长度、声明类型和 SHA-256。
3. 解析前复用供应商中立恶意文件扫描契约。
4. 在严格字节、行、列、值、嵌套、时间和内存限制下解析。
5. 通过所选不可变映射版本进行规范化。
6. 校验必填字段、结构化 Payload 规则、规范引用、证据引用和工作区归属。
7. 构建精确匹配去重结果和不可变 Dry-run Change Set。
8. 等待必需的人工或策略审批，不占用 HTTP 请求。
9. 通过业务事实应用服务分批进行有界事务应用。
10. 每个已接受规范版本在同一事务中写入审计和事务 Outbox 记录。
11. 生成脱敏结果报告，更新入驻完整度，并按保留策略删除临时来源/暂存数据。

Activity 具有有界超时、分类重试策略、Heartbeat 和明确取消点。Workflow 输入只含 ID、版本、哈希和安全摘要，不包含完整来源文件或 Secret。LangGraph 不用于确定性解析、校验、去重或应用。未来的映射建议只能把 LangGraph 作为可审查助手，绝不能静默接受映射或合并。

## 8. 校验与去重

校验是确定且带版本的。它复用规范业务事实校验器，并增加导入专用结构、引用、限制和映射校验。

去重顺序固定为：

1. 显式规范对象 ID，且必须属于当前工作区并匹配对象类型；
2. 精确规范化 `(workspace_id, object_type, slug)` 身份；
3. 只有对象 Schema 正式定义后才使用对象专属稳定标识，如产品 SKU 或市场 ISO 代码；
4. 其他情况一律生成冲突，交由所有者解决。

模糊、语义或模型建议的匹配绝不自动合并。当前版本冲突、引用缺失、匹配歧义或跨工作区 ID 均关闭式失败。同一文件内的重复行要确定性处理，并在报告中可见。

## 9. 应用、回滚与补偿

导入不会覆盖可变业务事实，而是创建新草稿对象或不可变版本。操作键由 `(workspace_id, import_job_id, source_row_id, change_set_version)` 派生，因此工作流重试不会重复创建已接受版本。

本基线不虚假承诺跨长时工作流的单一数据库回滚。Apply 前取消不会影响规范数据。Apply 期间，每个有界批次原子提交；后续批次失败时，作业进入 `partially_completed` 并明确显示已接受行和失败行。

补偿受策略治理且可审计：

- 新建、仍为 draft、且未被引用的对象，可以依据批准的保留策略归档或移除；
- 已 active 或产生下游引用的导入版本，只能通过新版本或归档状态纠正，绝不改写历史；
- 补偿会核验预期当前版本，遇到并发所有者编辑即停止；
- 审计事件、引用和不可变版本血缘始终保留。

## 10. 权限与审批

P2-D 引入窄权限，不能把 `business_truth.write` 当作无限批量权限：

- `workspace.onboarding.read`、`workspace.onboarding.write`、`workspace.onboarding.activate`；
- `business_truth.import.read`、`business_truth.import.create`、`business_truth.import.map`、`business_truth.import.apply`、`business_truth.import.cancel`、`business_truth.import.compensate`。

创建和校验作业的风险低于应用。应用 draft 创建/版本还要求 `business_truth.write`。应用 active 版本或激活资料必须通过策略审批，并具备相应高权限。补偿和冲突解决是高风险操作，必须记录操作者、原因、已审批 Change Set 哈希以及受影响规范版本。

首期拒绝账号、联系人、线索、同意、拒收、客户、商业和收入记录。这些数据必须在后续阶段使用其所属业务域的权限、隐私规则和专用导入契约。

## 11. 安全与隐私评审

- API 忽略或拒绝来源普通字段中提供的租户、操作者、状态、审计和存储标识；这些值只能来自可信上下文。
- 私有来源对象使用工作区范围 Key、精确供应商版本、加密策略、恶意文件扫描和短时授权，不能因此自动成为公开 Asset。
- 预览和导出问题报告会中和 CSV 公式前缀。任何公式、宏、HTML、脚本、模板表达式或转换代码都不会执行。
- 解析器拒绝无效编码、重复或超大表头、路径穿越、压缩包、嵌入文件、过度嵌套、解压技巧和不支持的内容类型。
- URL 始终只是惰性字符串，Worker 不会抓取。非可信内容绝不进入系统 Prompt 或工具指令。
- 日志、Trace、审计摘要和错误不包含原始来源行或疑似 Secret。报告使用稳定错误码和有界脱敏样本。
- 来源和暂存数据在终态后默认保留 30 天，且可按策略配置。审计、不可变版本血缘和必需合规证据遵循工作区保留策略。
- 速率限制和工作区配额覆盖活动作业、字节、行数、校验工作量和暂存保留量。资源耗尽时关闭式失败，不得削弱其他租户。
- 删除必须遵守 Legal Hold、策略、精确对象版本和审计要求。运维人员绝不能按未解析前缀或宽泛工作区路径删除。

## 12. 入驻激活门

激活是显式、带版本且与导入完成分离的操作。门槛要求：

- 已配置的必需规范对象类型齐全，且没有阻断性校验缺口；
- 每项必需事实具备 active 或显式批准的版本；
- 规范引用有效，且所需精确证据引用齐全；
- 工作区语言、时区、币种、负责人和基础策略已设置；
- 所有者审查剩余 Warning 和精确资料快照；
- 具备 `workspace.onboarding.activate` 并通过策略审批。

激活成功会发送 `BusinessProfileActivated`，Payload 包含精确对象版本 ID 和策略版本。重新打开入驻会创建新的激活候选，不修改旧激活记录。

## 13. 交付切片

### P2-D1 — 契约与隔离

新增模型、迁移、RLS、权限、API Schema、存储/扫描复用契约、幂等作业创建、状态、取消和测试。本切片不应用业务行。

实现证据：迁移 `0008` 定义了全部 7 张租户表、强制工作区 RLS 和 9 项分风险权限。版本化 API 支持持久化入驻创建/读取，以及导入创建、列表、读取、完成和取消；受约束的私有上传会进入专用 Temporal 核验与恶意文件扫描工作流。解析、映射、校验、Change Set 审查、激活和业务数据 apply 仍不属于 P2-D1。

### P2-D2 — 映射与校验

新增 CSV 与 Grovello JSON 适配器、不可变映射、有界暂存、预览、确定性校验、精确去重、问题报告，以及契约/安全测试。

实现证据：迁移 `0010` 在不改变规范业务事实的前提下，记录选定的不可变映射、校验幂等键、
解析器版本和专用 Temporal 工作流血缘。Worker 只接受显式指定允许分隔符的有界 UTF-8 CSV，
或 Manifest 与已授权作业一致的版本化 Grovello JSON 包；会拒绝畸形或重复 Schema，以及超出
行、列、标量值和嵌套限制的来源。系统仅执行白名单内的确定性转换，复用规范 Payload 与
工作区引用校验，并暂存内容哈希、精确身份结果、已中和公式的预览和仅含稳定错误代码的脱敏
问题。已实现的匹配信号仅包括规范 ID、精确规范化 slug、产品 SKU 和市场国家代码；信号冲突
与模糊相似绝不会自动合并。P2-D2 不创建 Change Set、规范对象、对象版本、激活或 apply 副作用。

### P2-D3 — 应用与入驻激活

新增 Dry-run Change Set、策略/审批交接、Temporal apply/取消/补偿、业务事实写入、资料完整度、激活、审计/Outbox 血缘和恢复测试。

实现证据：迁移 `0011` 新增工作区隔离的不可变 Change Set 操作、精确期望/结果版本血缘、apply 与补偿工作流标识、审批决定幂等以及激活快照。版本化 API 现可创建和读取 Dry-run 计划、记录绑定策略版本的审批决定、启动有权限的 apply 与补偿工作流、通过 Temporal 原生取消停止进行中的 apply，并且只激活完整的 active 资料。Apply 复用规范业务事实服务，操作键由工作区、作业、来源行和 Change Set 版本派生。补偿通过新建纠正版本或归档版本保留历史，并在检测到所有者并发新版本时停止。`BusinessProfileActivated` 包含精确对象版本 ID 与策略版本。PostgreSQL 升级/降级/再升级、强制 RLS、Alembic 元数据一致性、权限、工作流重试和恢复契约均已验证。在 P2-D4 双语操作旅程和完整本地多服务验收完成前，P2-D3 仍标记为 `foundation`。

### P2-D4 — 双语操作体验

通过薄 BFF 新增本地化入驻清单和导入工作台，覆盖加载、空、部分、失败、未授权、审批、重试、取消、补偿、移动端、键盘和屏幕阅读器状态。使用 PostgreSQL、Temporal、S3-compatible storage 和恶意文件扫描器完成真实本地端到端验证。

实现证据：“品牌与市场”现已新增高级且非主导航的 `/brand/business-setup` 与 `/brand/imports` 路由，通过服务端 BFF 提供英文和简体中文操作旅程，同时保持六个主导航入口不变。界面接通持久化入驻、受约束浏览器上传、持久化状态轮询、不可变映射、确定性校验证据、Change Set 审查、审批/apply 操作、取消、补偿和精确激活门禁；加载、空、不可用、未授权、冲突、阻塞、完成与已补偿状态均具备响应式和可访问控件。真实浏览器验收使用 PostgreSQL、Temporal、带最小权限应用账号的私有版本化 MinIO Bucket 和 ClamAV，将一个 CSV 从上传推进到干净扫描、映射、校验、Change Set 创建、规范 apply 与补偿，并产生 9 条审计记录和 9 条事务 Outbox 事件。全新工作区集成验收也已通过精确激活与补偿。由于生产身份、通用审批工作流、生产运维手册和非开发部署证据仍不在本次验收范围内，P2-D 继续标记为 `foundation`，不能标记为 `operational`。

## 14. 退出门与能力边界

只有全新工作区能上传允许的结构化来源，完成映射与校验，审查不可变 Change Set，幂等应用规范事实草稿，从部分失败恢复，生成审计/Outbox 血缘，满足入驻门槛，并在两种语言中激活精确企业资料快照，P2-D 才算完成。

本地退出证据现已齐备。剩余能力边界为：

- 入驻与导入只能标记为 `planned` 或 `foundation`，不能标记为 `operational`；
- 解析器演示或文件上传不等于受治理导入；
- 导入完成不代表资料已激活；
- 不声称已具备外部网盘、供应商同步、XLSX、AI 自动映射、模糊合并、客户数据导入或生产身份能力。

## 15. 架构、安全与交付评审结论

本基线符合现有架构：Next.js 仍是薄 BFF，FastAPI 负责业务事实，Temporal 负责持久化执行，PostgreSQL 保持权威，对象存储为私有且供应商中立，LangGraph 不参与确定性决策。它不改变十个业务域，也不引入供应商依赖。本基线无需 ADR。

若实现提议新增业务事实来源、新增一级业务域、允许可执行用户映射、由模型控制合并/应用、绕过扫描的来源路径，或改变正式技术基线，则必须停止并提交 ADR，等待产品所有者批准。
